The problem

Inclusion gets treated as a cost center. So it's the first thing cut.

It lives with a specialist. It shows up as a checklist after the design is done, or a ticket filed after the sprint is scoped. That's what makes it feel like a tax — and it's exactly why, when priorities tighten, this work is so often the first line item to go.

The fix isn't a bigger checklist or a later-stage audit. It's moving the same thinking to the point where a different word choice costs nothing — while the problem is still being framed, while the spec is still being written. Applied there, it isn't compliance. It's just better product thinking, on time.

Most "inclusive design" tools review work that's already finished. That makes them a left-shifted auditor — earlier, but still an audit. This kit is built to never produce a finished draft plus a separate list of what's missing.

How it works

Two modes. One rule: build it in, don't bolt it on.

Create mode

Co-drafts the work you're already doing

You're writing a problem statement, a PRD, a user story. This mode drafts alongside you — asking the questions that surface exclusion while a sentence is being written, so the inclusion lens ends up inside the sentence, not attached after it.

Learn mode

Explore the field, or find what nobody's building

No task, no deadline — just exploring inclusive design as a discipline, or using a real constraint as a generative prompt to surface opportunities nobody's addressing yet.

See the actual difference

"Users can reset their password by clicking the link in their confirmation email."
  • ⚑ Missing: what if the user can't access email (shared device, no account)?
  • ⚑ Missing: is this readable at 200% zoom or with a screen reader?
  • ⚑ Missing: what's the fallback for someone without reliable email access?

Flagged after the fact. The draft has to be reopened, the flags re-prioritized against everything else in the sprint — this is the pattern that turns inclusion into a tax.

"Users can reset their password via a link sent to their confirmation email, or via a phone-verified code for accounts without reliable email access — both paths readable at 200% zoom and compatible with screen readers."

Written this way the first time. Nothing to flag, nothing to reopen — the requirement was already complete.

Where it fits

Wherever product decisions get made.

Not a tool for one role on the team. Anywhere a problem gets framed, a requirement gets written, or a flow gets reviewed, this fits into the drafting itself.

01

Drafting the first version of a problem statement — before a roadmap or research plan gets built on top of it.

02

Writing acceptance criteria before a sprint starts — so "done" already includes it, instead of a ticket competing for priority later.

03

Reviewing a flow before it ships — catching what a specific, real constraint would break, while there's still time to change it cheaply.

Why this exists

Built by a product leader with hands-on experience and subject-matter depth in inclusive product design — not a framework applied from the outside.

I've been on teams where, when priorities tighten, this work is the first thing cut — not because it lacks value, but because it isn't the obvious line to revenue. I think that's backwards. The impact is real, and it compounds. This kit is one way to make the work harder to cut: build it into the fabric of how a spec gets written, not into a program that has to be re-funded every year.
